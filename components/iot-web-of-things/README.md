# Components: IoT and Web of Things (WoT)

The intersection of W3C Solid and the W3C Web of Things (WoT) is critical for decoupling IoT devices from centralized, proprietary data silos. This approach allows users to maintain direct, personal control over the data generated by their physical devices.

## 1. W3C Solid and Web of Things Integration
Recent literature and projects successfully connect these two W3C paradigms by using WoT to interface with hardware and Solid to store the resulting data securely.

### The WoT-Solid Project
A prominent example is the WoT-Solid project, which bridges the gap between heterogeneous wearable health devices and user-controlled data storage.
- **The Application:** An Android application that utilizes WoT standards to discover and interface with various Bluetooth Low Energy (BLE) wearables out-of-the-box.
- **The Integration:** Instead of pushing collected telemetry to a manufacturer's cloud, the system translates the device data into interoperable Resource Description Framework (RDF) formats and stores it directly into the user's Solid server (a Pod).
- **Open Source:** Available on GitHub under the repository `derwehr/WoT-Solid`.

### Autonomous Agents on the Web
The synergy between Solid (Social Linked Data) and WoT has sparked interest in decentralized multi-agent systems. The W3C Autonomous Agents on the Web (WebAgents) Community Group focuses on how this convergence allows automated software clients to navigate, query, and act upon a uniform hypermedia fabric.

## 2. Ontologies in Use
To ensure that data transitions seamlessly from a physical sensor to a decentralized Pod while remaining fully machine-readable, researchers rely on a combination of established Semantic Web ontologies:

- **WoT Thing Description (TD) Ontology:** The entry point and core semantic model of the WoT. It provides metadata and network-facing interfaces (Interaction Affordances) of physical devices, abstracting away underlying hardware protocols before data collection.
- **SOSA/SSN (Sensor, Observation, Sample, and Actuator) Ontology:** Used to translate collected data into a semantic structure suitable for a Solid Pod. It guarantees that an IoT metric (e.g., a heart-rate reading) is stored as standardized RDF, annotated with context like time and location, rather than an application-specific JSON string.

## 3. Access Control for Sensitive IoT Data
Solid handles access control and permissions for sensitive data like IoT health metrics using granular, policy-based systems.

### Access Control Policies (ACP)
ACP is the current standard for expressive, fine-grained management, ideal for complex authorization logic in IoT environments, replacing the legacy Web Access Control (WAC).
- **Policies and Matchers:** ACP functions through an authorization graph. A resource's Access Control Resource (ACR) links to a set of Policies. Each Policy contains Matchers (conditions) that must be met.
- **Granularity:** ACP can be applied to specific resources and containers within a Pod, allowing for policies based not just on "who" is accessing the data, but "under what conditions".
- **Contextual Access:** Matchers can evaluate the agent's WebID, the client application’s identifier, and other attributes to implement "purpose-based" access (e.g., granting read access to specific medical observations based on verifiable medical credentials).

### Application to IoT Architecture
Standard practices to enforce Least Privilege when integrating IoT hardware with Solid include:
- **Container Hierarchy:** Storing IoT data in nested Linked Data Platform (LDP) containers that mirror the device's architectural composition.
- **Resource-Level Policies:** Governing distinct data streams (e.g., diagnostic logs vs. user health metrics) within the same container using independent policies.
- **Separation of Duties:** Using `acl:Append` to allow IoT devices to stream data into a Pod without granting rights to modify or delete historical records, ensuring data audit trail integrity.

This architecture decouples authorization from hardware, allowing users to instantly revoke third-party application access to personal sensor data.
